# Field-Calibrated Benchmark Corpus Design

## Context

Phase 1 (richer fault modes + measurement memory) shipped on 2026-04-23.
Accuracy gates reported top-1 = 100% on three modes (dead / anomalous / hot)
against a 120-scenario corpus regenerated by `scripts/gen_hypothesize_benchmarks.py`.

That corpus is **auto-referential**: the generator runs `_simulate_failure()`
to produce each cascade, then samples observations from that cascade; the
solver re-runs the same `_simulate_failure()` on those observations and
ranks candidates. The 100% is a self-consistency check, not a measure of
real-world diagnostic accuracy. The solver has never been evaluated against
a repair the solver itself did not generate.

Separately, 9 MNT Reform community repair cases were crawled before Phase 1
began. Only 1 of 9 was encodable with the v1 binary `{dead, alive}` vocabulary
— Phase 1 extended to 4 modes (dead / anomalous / hot / shorted) and the
remaining 8 are now expressible in principle, but they were never encoded
as scenarios or run against the solver. We have no signal on whether Phase 1
actually helps a microsolder tech in the field.

This spec closes that gap. It captures ground truth from live repair
sessions AND seeds a small initial corpus from the 9 documented MNT cases,
then surfaces field accuracy as a distinct CI gate so regressions on real
repairs can't be masked by synthetic-corpus wins.

## Goal

Ship a feature that:

1. Persists every `hypothesize()` call during a live repair session to
   `memory/{slug}/repairs/{repair_id}/diagnosis_log.jsonl` with the
   observations snapshot and ranked output.
2. Adds a « Fixed » button to the repair dashboard that triggers a
   validation dialogue with Claude — Claude summarises the session's
   fixes and calls a new `mb_validate_finding` tool to persist the ground
   truth.
3. Stores validated outcomes in `memory/{slug}/repairs/{repair_id}/outcome.json`.
4. Introduces a `data/historical_cases.json` file (git-tracked, manually
   curated) seeding the corpus with the 9 MNT Reform cases.
5. Provides `scripts/build_benchmark_corpus.py` that unions live outcomes
   + historical cases into a field-real scenarios JSONL at
   `tests/pipeline/schematic/fixtures/hypothesize_field_scenarios.json`.
6. Adds a parallel CI gate file `test_hypothesize_field_accuracy.py` with
   per-mode thresholds calibrated on the field corpus (initially
   permissive; will tighten as the corpus grows).

The demo flow:

> Tech opens MNT Reform repair, measures `+3V3 = 0.02 V`, asks Claude.
> Claude calls `mb_hypothesize`, returns U12 top-1. diagnosis_log logs
> the obs + ranking. Tech replaces U12, confirms « ça marche ». Tech
> clicks « Fixed » on dashboard. WS event triggers a validation chat:
> Claude reads the session, proposes « Le fix semble être U12 (dead,
> remplacé). Correct ? ». Tech confirms. Claude calls
> `mb_validate_finding`. outcome.json written. Next
> `build_benchmark_corpus.py` run picks this up, field accuracy for
> dead mode gets one more real data point.

## Non-goals

- **Phase 5 numeric scoring** (proximity term on measurements). Still
  deferred. This spec builds the calibration substrate it will need, not
  the scoring change itself.
- **USB-HID multimeter integration.** Orthogonal. The validation flow
  works identically whether measurements come from typing or from a HID
  bridge.
- **Auto-detection of fixes via chat NLU without the button.** The
  trigger stays explicit: tech clicks « Fixed » on the dashboard. Claude
  only performs the dialogue once a validation is actively requested.
  Passive chat detection (« j'ai remplacé X et ça marche ») is too
  fragile to rely on as a ground-truth source.
- **Historical case ingestion beyond the 9 MNT Reform cases.** iFixit /
  Rossmann / forum crawling is out of scope. If the dataset grows beyond
  the initial seed, it grows via live validated repairs.
- **Replaying historical diagnosis_log.** Historical cases skip the
  per-call ranking trace — they have a single `{observations,
  ground_truth}` pair, no evolution over time. Only live sessions
  capture the trace.

## Architecture

Five units, each with one responsibility:

1. **Diagnosis log** (`api/tools/hypothesize.py` extension, ~30 LOC)
   - After `mb_hypothesize` computes its result, append one JSON line to
     `memory/{slug}/repairs/{repair_id}/diagnosis_log.jsonl` when a
     `repair_id` is in scope. Entry shape: `{timestamp, observations,
     hypotheses_top5, pruning_stats}`.
   - Silent no-op when no `repair_id` (legacy flow, tech not in a repair
     session).
   - Errors swallowed and logged — same best-effort contract as the
     measurement journal.

2. **Validation trigger** (`web/js/dashboard.js` + `api/main.py` WS handler, ~80 LOC)
   - New DOM button `#dashboardFixBtn` in the repair dashboard (topbar
     session pill area). Label: « Marquer fix ».
   - On click, the frontend sends a JSON payload
     `{type: "validation.start", repair_id}` through the existing
     `/ws/diagnostic/{slug}` socket.
   - The diagnostic runtime (`runtime_direct.py` + `runtime_managed.py`)
     intercepts this event and injects a synthesised user-role message
     into the agent's next turn: « [Action tech — Marquer fix]
     L'utilisateur vient de confirmer que la repair {repair_id} est
     résolue. Relis l'historique du chat + les mesures récentes, résume
     en une phrase les composants remplacés / réparés, puis appelle
     `mb_validate_finding` avec les `fixes` confirmés. Si ambigu, demande
     une clarification au tech avant d'appeler l'outil. ».
   - The message is tagged `role="user"` with a `source: "trigger"`
     marker in the JSONL chat history so the UI can style it as a
     system-event bubble (or filter it out of the visible chat
     transcript). This keeps the agent's turn model unchanged — no
     custom system-augment channel needed — while preserving full
     audit trail.

3. **`mb_validate_finding` tool** (`api/tools/validation.py`, new, ~120 LOC)
   - Signature: `mb_validate_finding(device_slug, repair_id, memory_root,
     fixes: list[dict], tech_note: str | None = None) -> dict[str, Any]`
   - Each fix: `{refdes, mode, rationale: str}`. Mode literal includes
     the 4 PhaseMode 1 values plus a new `passive_swap` placeholder
     (unused in Phase 1 scoring, reserved for Phase 4's passive
     injection).
   - Writes `memory/{slug}/repairs/{repair_id}/outcome.json` as:

     ```json
     {
       "validated_at": "2026-04-23T19:45:12Z",
       "repair_id": "0f8ab295e689",
       "device_slug": "mnt-reform-motherboard",
       "fixes": [
         {"refdes": "U12", "mode": "dead", "rationale": "buck mort, +3V3 absent, remplacé, reflow OK"}
       ],
       "tech_note": null,
       "agent_confidence": "high"
     }
     ```

   - Also emits a WS event `simulation.repair_validated` so the dashboard
     can flip the repair to a « validated » visual state.
   - Validated refdes checked against the current board / electrical
     graph — unknown refdes rejected with `{validated: false, reason:
     "unknown_refdes"}`, same guardrail pattern as other mb_* tools.

4. **Historical cases file** (`data/historical_cases.json`, new, hand-curated)
   - Shape:

     ```json
     [
       {
         "id": "mnt-reform-u13-buck-dead",
         "device_slug": "mnt-reform-motherboard",
         "source": "https://source-url",
         "description": "Carte ne démarre pas, +1V2 absent au multimètre, U13 (buck) mort suite à surtension. Remplacement + reflow = fix.",
         "observations": {
           "state_rails": {"+1V2": "dead"},
           "state_comps": {},
           "metrics_rails": {"+1V2": {"measured": 0.025, "unit": "V", "nominal": 1.2}}
         },
         "ground_truth_fixes": [
           {"refdes": "U13", "mode": "dead"}
         ]
       },
       ...
     ]
     ```

   - Seed contains the 9 MNT Reform cases. Manually saisi, reviewed by
     Alexis for accuracy before commit.
   - File is version-controlled — corpus evolution is trackable through
     git history. No auto-generation / auto-ingestion.

5. **Corpus builder** (`scripts/build_benchmark_corpus.py`, new, ~200 LOC)
   - CLI: `python scripts/build_benchmark_corpus.py --out
     tests/pipeline/schematic/fixtures/hypothesize_field_scenarios.json`
   - Scans every `memory/*/repairs/*/outcome.json`. For each, joins with
     the matching `measurements.jsonl` (latest measurement per target
     becomes the observations snapshot) to produce a scenario at the
     same shape as the synthetic corpus.
   - Appends every entry from `data/historical_cases.json` with a
     `source: "historical"` marker. Live entries get `source: "live"`.
   - Emits a single JSONL-ish JSON array so the existing accuracy-test
     loader can consume it directly.
   - Run as a pre-test step in CI (hook in `make test-field` target) OR
     committed manually after every validated repair (same pattern as
     `scripts/gen_hypothesize_benchmarks.py` today — file is
     version-controlled, diff shows drift).

6. **Field accuracy gates** (`tests/pipeline/schematic/test_hypothesize_field_accuracy.py`, new, ~120 LOC)
   - Mirror of `test_hypothesize_accuracy.py` but reads the
     field-scenarios fixture.
   - Per-mode thresholds start VERY permissive (top-1 ≥ 30%, top-3 ≥
     50%, MRR ≥ 0.40 for all modes) because the corpus is tiny (9
     seeded + however many live repairs exist at merge time).
   - Thresholds tightened manually as the corpus grows — explicit
     commit, no auto-calibration to avoid silent threshold drift.
   - Skips per-mode if fewer than 3 scenarios for that mode (small-N
     statistical honesty).

Cross-cutting: the live diagnosis_log + outcome capture DOES NOT break
any existing flow. Tests that don't touch the new files continue to
pass. The feature is purely additive on the backend. The frontend adds
one button but the existing dashboard layout is preserved.

## Data shapes (Pydantic v2, all `ConfigDict(extra="forbid")`)

```python
# api/agent/diagnosis_log.py (new)

class DiagnosisLogEntry(BaseModel):
    model_config = ConfigDict(extra="forbid")

    timestamp: str                    # ISO 8601 UTC
    observations: dict                # raw Observations.model_dump()
    hypotheses_top5: list[dict]       # [{kill_refdes, kill_modes, score, narrative}]
    pruning_stats: dict               # {single_candidates_tested, wall_ms, ...}


# api/agent/validation.py (new)

class ValidatedFix(BaseModel):
    model_config = ConfigDict(extra="forbid")

    refdes: str
    mode: Literal["dead", "alive", "anomalous", "hot", "shorted", "passive_swap"]
    rationale: str


class RepairOutcome(BaseModel):
    model_config = ConfigDict(extra="forbid")

    validated_at: str
    repair_id: str
    device_slug: str
    fixes: list[ValidatedFix]
    tech_note: str | None = None
    agent_confidence: Literal["high", "medium", "low"] = "high"
```

```python
# scripts/build_benchmark_corpus.py

class HistoricalCase(BaseModel):
    model_config = ConfigDict(extra="forbid")

    id: str
    device_slug: str
    source: str
    description: str
    observations: dict               # {state_comps, state_rails, metrics_comps, metrics_rails}
    ground_truth_fixes: list[dict]   # [{refdes, mode}]


class FieldScenario(BaseModel):
    model_config = ConfigDict(extra="forbid")

    id: str
    slug: str
    source: Literal["live", "historical"]
    observations: dict
    ground_truth_kill: list[str]
    ground_truth_modes: list[str]
    description: str | None = None   # for historical cases only
```

## HTTP + WS surface

No new HTTP endpoints — validation flows entirely through the existing
`/ws/diagnostic/{slug}` socket.

New WS events:

```
client → server
  {type: "validation.start", repair_id: "..."}

server → client (emitted by mb_validate_finding)
  {type: "simulation.repair_validated", repair_id, fixes_count}
```

The existing `simulation.observation_set` / `observation_clear`
envelopes are untouched.

## Frontend changes

### Dashboard (`web/js/dashboard.js`)

Add a button inside the repair session pill area of the topbar:

```html
<button id="dashboardFixBtn" class="btn-primary" title="Marque la repair comme résolue — Claude valide et enregistre les fixes">
  ✓ Marquer fix
</button>
```

Click handler:

```javascript
document.getElementById("dashboardFixBtn").addEventListener("click", () => {
  const repairId = getCurrentRepairId();
  if (!repairId) return;
  const ws = window.__diagnosticWS;
  if (!ws || ws.readyState !== WebSocket.OPEN) {
    toast("Pas de session agent active.");
    return;
  }
  ws.send(JSON.stringify({type: "validation.start", repair_id: repairId}));
});
```

The button is visible only when a repair is active (same gate as the
existing session pill). When `simulation.repair_validated` fires, swap
the button to « ✓ Validé » disabled state.

### LLM handler (`web/js/llm.js`)

Listen for `simulation.repair_validated` events, update the dashboard
button state, optionally show a toast.

## Runtime agent changes

### `runtime_direct.py` and `runtime_managed.py` (~50 LOC each)

Before dispatching the next agent turn, check for pending
`validation.start` events received since the last turn. When present:

- Prepend a system-level augment to the turn's context:
  « [VALIDATION REQUESTED] Le tech vient de cliquer 'Marquer fix' pour
  la repair {repair_id}. Lis l'historique du chat + les mesures
  récentes, résume en une phrase les composants remplacés / réparés,
  puis appelle `mb_validate_finding` avec les `fixes` confirmés. Si
  ambigu, demande clarification au tech avant d'appeler l'outil. »

- Consume the event (don't re-inject on the next turn).

The augment is injected as a system-role message with a marker so
`chat_history.jsonl` records it but the UI chat panel can optionally
filter it out (distinguishing from user messages).

### `api/tools/validation.py` (new)

```python
def mb_validate_finding(
    *,
    device_slug: str,
    repair_id: str,
    memory_root: Path,
    fixes: list[dict],
    tech_note: str | None = None,
    agent_confidence: str = "high",
) -> dict[str, Any]:
    # Validate each fix's refdes against the electrical_graph.json /
    # board.json if present — reject with {validated: false,
    # reason: "unknown_refdes", invalid: [...]} on mismatch.
    # Write RepairOutcome to outcome.json.
    # Emit simulation.repair_validated WS event.
    # Return {validated: True, repair_id, fixes_count}.
```

Registered in `api/agent/manifest.py` alongside the existing `mb_*`
tools.

## Corpus builder

```python
# scripts/build_benchmark_corpus.py (sketch)

def build_live_scenarios(memory_root: Path) -> list[dict]:
    from api.agent.measurement_memory import synthesise_observations
    scenarios = []
    for outcome_path in memory_root.glob("*/repairs/*/outcome.json"):
        outcome = RepairOutcome.model_validate_json(outcome_path.read_text())
        observations = synthesise_observations(
            memory_root=memory_root,
            device_slug=outcome.device_slug,
            repair_id=outcome.repair_id,
        )
        scenarios.append({
            "id": f"live-{outcome.device_slug}-{outcome.repair_id}",
            "slug": outcome.device_slug,
            "source": "live",
            "observations": observations.model_dump(),
            "ground_truth_kill": [f.refdes for f in outcome.fixes],
            "ground_truth_modes": [f.mode for f in outcome.fixes],
        })
    return scenarios


def build_historical_scenarios(historical_path: Path) -> list[dict]:
    cases = json.loads(historical_path.read_text())
    return [
        {
            "id": f"hist-{c['id']}",
            "slug": c["device_slug"],
            "source": "historical",
            "observations": c["observations"],
            "ground_truth_kill": [f["refdes"] for f in c["ground_truth_fixes"]],
            "ground_truth_modes": [f["mode"] for f in c["ground_truth_fixes"]],
            "description": c["description"],
        }
        for c in cases
    ]
```

Output file shape is a flat JSON array of FieldScenario. The
accuracy-test loader filters by `ground_truth_modes[0]` for per-mode
gates.

## Field accuracy test (sketch)

```python
# tests/pipeline/schematic/test_hypothesize_field_accuracy.py

FIELD_FIXTURE = Path(__file__).parent / "fixtures" / "hypothesize_field_scenarios.json"

# Permissive starting thresholds — corpus small, tighten as it grows.
FIELD_THRESHOLDS = {
    "dead":      {"top1": 0.30, "top3": 0.50, "mrr": 0.40},
    "anomalous": {"top1": 0.25, "top3": 0.45, "mrr": 0.35},
    "hot":       {"top1": 0.30, "top3": 0.50, "mrr": 0.40},
    "shorted":   {"top1": 0.20, "top3": 0.40, "mrr": 0.30},
}
MIN_SCENARIOS_PER_MODE = 3   # skip below this


@pytest.mark.parametrize("mode", ["dead", "anomalous", "hot", "shorted"])
def test_field_top1_per_mode(mode: str):
    records = [r for r in _load_field_scenarios() if r["mode"] == mode]
    if len(records) < MIN_SCENARIOS_PER_MODE:
        pytest.skip(f"mode={mode}: only {len(records)} scenarios, need {MIN_SCENARIOS_PER_MODE}")
    top1 = sum(1 for r in records if r["rank"] == 1) / len(records)
    assert top1 >= FIELD_THRESHOLDS[mode]["top1"]
```

## Rollout plan (high level)

The implementation plan will split this into ~12-14 tasks. Structural arc:

1. **Live diagnosis log** (tasks 1-2) — `diagnosis_log.py` store,
   `api/tools/hypothesize.py` hook.
2. **Validation outcome shape + tool** (tasks 3-5) —
   `ValidatedFix` / `RepairOutcome` Pydantic, `mb_validate_finding`
   tool, manifest registration.
3. **WS envelope + runtime handler** (tasks 6-7) —
   `SimulationRepairValidated`, runtime `validation.start` handling +
   system augment injection.
4. **Dashboard button + WS wire** (task 8) — « Marquer fix » button,
   WS send, WS handler for validated state. Browser-verify required.
5. **Historical cases seed** (task 9) — hand-curate the 9 MNT cases
   into `data/historical_cases.json`. Requires Alexis input on
   sources / observations / fixes.
6. **Corpus builder** (task 10) — `build_benchmark_corpus.py`,
   Makefile target.
7. **Field accuracy gates** (tasks 11-12) — new test file,
   permissive starting thresholds. Run once after seed to establish
   baseline.
8. **Smoke + hand-off** (task 13) — full live demo (record, diagnose,
   fix, validate), field corpus rebuilt, field gates pass.

Each group lands in a single commit. Tests pass at every commit.

## Phased follow-ups (out of this spec)

- **Phase 5 numeric scoring on top of the field corpus** — the corpus
  built here is the calibration substrate for the `|measured -
  predicted| / nominal` proximity term. Tune the weight only once the
  field corpus has ≥ 30 validated scenarios.
- **Multi-device historical corpus.** iPhone X logic board,
  Demo-Pi, other packs can each accumulate their own `outcome.json`
  live entries. No change to the corpus builder required — the glob
  already spans all devices.
- **Agent self-critique on failed diagnoses.** When a live session's
  `diagnosis_log` shows the top-1 never matched the validated fix,
  surface this to Claude as a meta-signal (« sur cette repair tu as
  proposé X au top-1 mais le tech a confirmé Y — pourquoi ? »). Out
  of scope here, worth a follow-up spec.

## Open questions

All resolved during brainstorming. One decision locked post-review:

- **Validation trigger = explicit button, not passive chat detection.**
  Passive NLU on chat history is too fragile to seed a ground-truth
  dataset. The « Marquer fix » button gives an unambiguous signal and
  lets Claude orchestrate the confirmation dialogue explicitly.

## Dette backlog (out of scope entirely)

- Automatic historical-case ingestion from iFixit / forums. If the
  dataset needs to scale past manual curation, write a dedicated
  ingestion pipeline — separate product.
- Comparison of field vs synthetic accuracy across time (regression
  dashboard). Useful for tracking model drift; out of Phase 1's scope.
- Cross-device aggregate accuracy (normalised by device complexity).
  Nice-to-have reporting, not an engineering requirement.
